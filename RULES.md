# RULES.md — 運用ルール

このリポジトリで全メンバーが守るべき運用ルール。
`CLAUDE.md` は ClaudeCode 向け、本ファイルは **人間とエージェント両方** に対するルール。

---

## 1. フォルダ命名ルール

### 1-1. 既存命名を優先

新しいフォルダを作る前に、既存命名を確認すること。新規発明は最終手段。

### 1-2. 講座フォルダ命名

```
{言語}_{講座名}{ローマ数字}
```

例:
- `ベトナム語_よくわかる！ロジスティクス入門Ⅰ`
- `中国語_よくわかる！ロジスティクス入門Ⅳ`
- `個人情報の取扱`（言語が日本語のときは省略可）

### 1-3. UNIT 命名

```
UNIT{番号}
```

中身は `スライド{N}.png`（macOS APFS で NFD 形式になる点に注意）。

### 1-4. plck-main 側の unit_id 命名

```
{lang}-{course}-unit0{N}
```

例: `zh-logistics01-unit01`、`vi-logistics04-unit03`

### 1-5. ZIP 名

```
{lang}_{course}_unit0{N}.zip
```

ハイフン → アンダースコアに変換。例: `zh_logistics01_unit01.zip`。

### 1-6. 一時ファイル / 失敗作の置き場所

| 種類 | 配置先 |
|---|---|
| 作業中の一時ファイル | `tmp/`（`.gitignore` 対象） |
| 過去の失敗版 ZIP | `講座セットフォルダ/.../LMS搭載用ZIP/【旧・*失敗版】/` |
| 隔離が必要な一時生成物 | `archive/failed/`（`.gitignore` 対象） |

## 2. 画像ルール

### 2-1. 16:9 比率を維持

- スライド比率は 16:9 固定。元 PPTX のページサイズも 16:9 を保つ。
- スライド内のレイアウトを縦長 / 正方形に変えるのは禁止。

### 2-2. 元画像と LMS 用画像の対応関係を崩さない

- `講座セットフォルダ/.../UNIT{N}/スライド{M}.png` ⇔ `plck-main/contents/scenes/slide/{unit_id}/slide/{M}.png` は **SHA-256 完全一致** が必須。
- 差し替え時はバイナリ完全コピー（`cp` または `rsync`）を使う。Finder のドラッグ&ドロップでも可だが、メタ情報の差異がないか preflight で確認する。

### 2-3. 日本語講座で成立している UI を翻訳版でも維持

- ヘッダー色・タイトル行数・ハンバーガー有無などを翻訳版で勝手に変えない。
- 仕様は [LMS_BUILD_RULES.md](LMS_BUILD_RULES.md) §3、§5、`tools/README.md` §3 を参照。

### 2-4. 高画質 PNG 仕様

- 解像度: 2668×1500（200 DPI）以上
- 720×405 などの低画素 PNG は使用禁止
- 中国語 (zh-\*) ではデザイン担当出力 PNG をそのまま使用（再レンダリング禁止）

## 3. LMS 搭載用 ZIP ルール

### 3-1. 対象講座・対象 UNIT を明記して生成

- 単発生成のときは UNIT 名 / 言語 / コース名 を作業ログに残す（コミットメッセージで OK）。
- 24 本一括の場合は `./tools/scripts/build_all.sh` 1 コマンドで OK。

### 3-2. 生成前後でファイル数を確認

- 生成前: `ls 講座セットフォルダ/{講座}/LMS搭載用ZIP/*.zip | wc -l`
- 生成後: 同じ場所が **3 ZIP**（unit01〜03）になっていることを確認。

### 3-3. 古い ZIP との混在に注意

- 旧失敗版は必ず `【旧・*失敗版】/` に退避してから新版を配置。
- ファイル名で「v1 / v2」のような命名をしない（混乱の元）。

### 3-4. 再生成時は CHANGELOG に記録する

- `CHANGELOG.md` に「いつ」「どの講座」「なぜ再生成したか」を 1 行記録する。
- 例: `2026-04-21: zh-logistics01〜04 全UNIT 再生成（中国語フォント崩れ修正）`

### 3-5. ビルド前後の絶対ルール

- 必ず `./tools/scripts/build_all.sh` 経由で生成。手動 `npx plck build` で済ませない。
- preflight が FAIL した状態で ZIP 化しない。
- ビルド後は `verify_zip.sh` を通す（`build_all.sh` 内で自動実行）。

## 4. 多言語ルール

| 言語 | prefix | 既存の運用 | 追加時の確認事項 |
|---|---|---|---|
| 日本語 | `ja-` / `privacy-` / `security-` | 確立済み | 基準。仕様変更は全体に波及するため慎重に |
| ベトナム語 | `vi-` | ロジスティクス Ⅰ〜Ⅳ で運用中 | Meiryo→Arial 置換 OK・声調記号の分離を確認 |
| 中国語 | `zh-` | ロジスティクス Ⅰ〜Ⅳ で運用中 | **Meiryo→Arial 置換 禁止**・デザイン出力 PNG をそのまま使用 |
| 英語 | `en-` | 今後追加予定 | 言語タグ確認・括弧語 `（English）` 確認 |
| ポルトガル語 | `pt-` | 今後追加予定 | 言語タグ確認・括弧語 `（Português）` 確認 |
| 韓国語 | `ko-` | 今後追加予定 | CJK のため zh-\* と同様の警戒 |
| タガログ語 | `tl-` | 今後追加予定 | 鼻音記号・アクセントの分離検証 |

**新言語追加時の必須確認**:

1. 言語タグの管理画面登録
2. 括弧語表記の登録（例: `（中国語）` `（English）`）
3. 既存（英語・ポルトガル語で成立している）表示仕様が新言語にも反映されるか
4. ロジスティクス系の場合は **ロジスティクスのカテゴリタグ** が付いているか
5. preflight.sh / build_all.sh の `PAIRS` への追記

詳細は [docs/translation-rules.md](docs/translation-rules.md) と [docs/tag-rules.md](docs/tag-rules.md)。

## 5. Git / コミット運用ルール

### 5-1. 1 機能 1 修正 = 1 コミット

- 「画像差し替えと CSS 修正」を 1 コミットにまとめない。
- 大きな変更は分割してから commit。

### 5-2. コミットメッセージは「なぜ」を含める

例:

```
画像差し替え（vi-logistics03-unit02 スライド5）

理由: スライド5の図版に誤字があったため、デザイン担当からの差し替え PNG に更新。
preflight / verify_zip 通過済み。
```

### 5-3. push 先の確認

push する前に必ず:

```bash
git remote -v
gh auth status
```

を実行し、push 先がクロスサポートのアカウントになっていることを確認。

### 5-4. push 前は Codex レビューを必須

- 親 [CLAUDE.md](CLAUDE.md) §3 のルールに従う。
- 指摘事項は修正 → 再レビュー → push の順。

### 5-5. 危険な操作の禁止

| 操作 | 可否 | 備考 |
|---|---|---|
| `git push --force` to main | ❌ 禁止 | 例外時は PM 承認 |
| `git reset --hard` | ⚠️ 注意 | 未コミット変更が消える |
| `git add .` / `git add -A` | ⚠️ 非推奨 | 巨大ファイル誤混入の温床。ファイル名指定を推奨 |
| `git commit --amend` (push 後) | ❌ 禁止 | 共有履歴の改変 |
| `git commit --no-verify` | ❌ 禁止 | hook を意図的に飛ばすのは禁止 |

## 6. 禁止事項

このリポジトリで **絶対に行ってはいけない** こと:

1. 既存講座データの無断削除
2. 生成済み ZIP の無断上書き（必ず `【旧・*失敗版】/` に退避してから）
3. UI 構成の勝手な変更（ヘッダー色・タイトル行数・ハンバーガー有無 等）
4. 画像比率の変更（16:9 固定）
5. 日本語講座と翻訳講座で表示仕様を変えること
6. 個人 GitHub アカウントへの push
7. `.env` や認証情報の commit
8. 中国語（zh-\*）講座での Meiryo→Arial 置換
9. `tools/scripts/` 配下のスクリプトの破壊的変更（必要なら別ファイルとして追加）
10. preflight が FAIL したまま ZIP を本番配置すること

## 7. このルールの更新方法

1. 変更提案は PR で出す
2. PR 説明欄で「なぜ変更が必要か」を記述
3. PM 承認 → merge
4. `CHANGELOG.md` に変更内容を記録
